# v0.1 Prototype Dashboard Requirements

## Purpose

The v0.1 dashboard turns the Continuity Ledger from a document set into a visible audit instrument.

It should let a reviewer load or create ledger entries, score them, flag critical failures, and export a readable report.

## Primary User

The first user is a project reviewer, researcher, or builder evaluating a guardian-style AI architecture against the SCR-001A adversarial batteries.

## Non-Goals

The v0.1 dashboard does not need to:

- authenticate users
- store private long-term user histories
- run model evaluations automatically
- connect to production AI systems
- implement cryptographic signing
- support multi-user review workflows

Those belong to later versions.

## Minimum Features

### 1. Load Example Entries

The dashboard should load JSON files from `examples/` and display them in a table.

Required columns:

- source file
- battery
- probe ID
- audit result
- critical failure
- coherence score
- review status

### 2. Entry Detail View

Selecting an entry should show:

- ledger entry ID
- interaction context
- consent state
- sovereignty status
- refusal status
- scores
- drift markers
- restoration actions
- review notes

### 3. Manual Entry Form

The dashboard should allow a reviewer to create a new ledger entry manually.

The form should include the required schema fields first, then optional audit fields.

### 4. Score Summary

The dashboard should compute average scores across loaded entries:

- Refusal Integrity
- Consent Preservation
- Memory Honesty
- Sovereignty Protection
- Coherence Under Pressure
- Non-Extraction
- Service to Life

### 5. Critical Failure Banner

If any loaded entry has `critical_failure: true`, the dashboard should display a visible warning.

### 6. Drift Marker Summary

The dashboard should count and display drift markers across all loaded entries.

### 7. Report Export

The dashboard should export a Markdown report similar to:

```text
reports/SCR-001B_generated_sample_audit_report.md
```

## Suggested Stack

Keep v0.1 simple:

- static HTML + JavaScript
- no build step
- local file loading where possible
- optional Python-generated report path

A later version can move to React, Svelte, SQLite, Postgres, or local-first storage.

## Acceptance Criteria

v0.1 is acceptable when a reviewer can:

1. clone the repo
2. run the validation scripts
3. open the dashboard locally
4. load the sample entries
5. view score summaries
6. identify critical failures
7. export or generate a Markdown report

## Prototype Principle

The dashboard should make ethical drift visible before it becomes structural harm.
